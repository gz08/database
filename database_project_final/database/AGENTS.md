# 项目代理开发规则（AGENTS）

## 一、项目说明
本仓库是数据库课程大作业项目，题目为 **学生管理系统**，技术栈为：

- openGauss
- Python
- Streamlit

系统属于数据库应用系统，必须满足课程作业要求，而不是做成随意扩展的普通练手项目。

---

## 二、总体开发目标
开发一个基于 openGauss 的学生管理系统，面向以下三类角色：

- 系统管理员
- 教师/教务
- 学生

系统需至少覆盖以下功能：

- 登录
- 权限控制
- 学生信息管理
- 课程信息管理
- 选课管理
- 成绩管理
- 查询功能（包含模糊查询）

---

## 三、必须满足的课程要求
开发过程中必须确保以下内容被覆盖：

### 1. 题目形式
- 题目为自定义数据库应用系统
- 当前题目为学生管理系统

### 2. 数据库要求
- 必须基于 openGauss
- 数据设计必须规范、清晰、可说明

### 3. 小组要求
- 项目默认按 3-4 人小组作业标准考虑
- 文档中需要能够说明每位成员负责的模块

### 4. 功能要求
必须体现：

- 数据增加
- 数据删除
- 数据修改
- 数据查询
- 用户权限管理

### 5. 查询要求
- 查询功能不能只做精确查询
- 必须支持至少一种模糊查询方式

### 6. 提交要求
最终项目应支持整理出以下成果：

- 源代码
- 数据库文件
- 英文开发文档
- 演示视频（3分钟以内）

---

## 四、开发原则

### 1. 优先保证课程作业完成度
项目应优先满足：

- 可运行
- 可演示
- 可说明
- 可交付

不要为了“高级感”引入不必要复杂设计。

### 2. 优先最小可行实现
对于功能开发，优先采用：

- 结构清晰
- 代码易懂
- 便于课程答辩
- 便于文档编写

### 3. 不要过度拆分文件
当前仓库建议维持精简结构，不要擅自拆成大量小文件。

### 4. 不要随意更改核心表名
默认核心表如下：

- roles
- users
- students
- teachers
- courses
- enrollments
- grades

如果无明确要求，不要修改这些表名。

### 5. 不要随意改变角色模型
默认角色固定为：

- admin
- teacher
- student

不应擅自新增复杂权限体系。

当前版本角色权限口径固定为：

- admin：管理账号、分配角色、重置密码、启用/禁用账号、查看全局数据；负责学生信息新增/删除/修改
- teacher：仅查看学生信息；查看选课信息；管理课程；录入与修改成绩
- student：查看个人信息、查看课程、选课/退课、查看个人成绩

教师页面不提供学生信息 CRUD 界面。

## 六、文件职责约束

### 1. `schema.sql`
负责数据库表结构定义：

- 主键
- 外键
- 非空约束
- 唯一约束
- 必要的检查约束

### 2. `seed.sql`
负责初始化数据：

- 角色数据
- 管理员账号
- 测试教师账号
- 测试学生账号
- 少量测试课程、选课、成绩数据

### 3. `db.py`
负责数据库连接与基础执行逻辑。

### 4. `init_db.py`
负责执行 `schema.sql` 与 `seed.sql` 完成数据库初始化。

### 5. `models.py`
负责所有核心表的 CRUD 数据访问逻辑。

### 6. `services.py`
负责业务规则处理：

- 重复选课检查
- 成绩范围检查
- 数据合法性校验
- 权限辅助判断

### 7. `auth.py`
负责：

- 登录校验
- 密码处理
- 当前用户与角色读取
- 登录状态辅助逻辑

### 8. `pages_admin.py`
负责管理员页面。

### 9. `pages_teacher.py`
负责教师/教务页面。

### 10. `pages_student.py`
负责学生页面。

### 11. `utils.py`
负责通用工具逻辑，不要承载核心业务。

### 12. `app.py`
负责应用入口与页面分流。

## 七、数据库设计约束

### 1. 设计时必须体现关系型数据库规范
应合理使用：

- 主键
- 外键
- 唯一约束
- 非空约束

### 2. 数据设计应支撑权限控制
例如：

- `users` 关联 `roles`
- `students` / `teachers` 关联 `users`

### 3. 查询功能必须支持模糊查询
例如：

- 按学生姓名模糊搜索
- 按课程名称模糊搜索

### 4. 成绩与选课关系必须合理
成绩数据应基于选课记录，而不是随意脱离业务关系独立存在。

## 八、开发行为规则

### 1. 优先小改动
修改时尽量局部改动，不要大面积重构。

### 2. 不擅自新增依赖
如非必要，不要额外引入复杂第三方库。

### 3. 不混写代码层次
- 页面代码不要直接堆满 SQL
- 业务规则不要全部塞进页面文件
- 数据访问逻辑不要和界面渲染混成一团

### 4. SQL 必须参数化
避免直接字符串拼接 SQL。

### 5. 保持代码可读性
变量命名清晰，函数职责单一，不做炫技式写法。

## 九、验证要求
每完成一个阶段，至少验证以下内容：

### 数据库阶段
- 表是否成功创建
- 外键关系是否正确
- 初始化数据是否导入成功

### 登录阶段
- 管理员可登录
- 教师可登录
- 学生可登录
- 不同角色能看到不同页面

### 业务阶段
- 学生信息增删改查可用
- 课程信息增删改查可用
- 选课/退课可用
- 成绩录入与查询可用
- 模糊查询可用

### 权限阶段
- 学生不能管理账号
- 学生不能修改他人成绩
- 教师不能管理系统角色
- 教师只能查看学生信息，不能新增/删除/修改学生信息
- 管理员可启用/禁用账号
- 管理员不能通过学生页面执行学生操作

## 十、完成标准
一个任务完成，至少要满足：

- 相关代码已写入对应文件
- 没有明显语法错误
- 对应功能可运行
- 不破坏现有角色分流
- 与课程要求一致
- 后续可用于整理文档和演示视频

## 十一、禁止事项
- 不要为了“高级”而过度设计
- 不要擅自改动项目核心结构
- 不要把所有代码堆进一个文件
- 不要忽视权限控制
- 不要只做精确查询
- 不要遗漏文档与交付材料要求
- 不要开发与课程评分无关的大量附加功能

## 十二、最终目标
该项目不是为了做成完整商业系统，而是为了高质量完成数据库课程大作业。

开发重点应始终围绕：

- 课程要求覆盖完整
- 数据库设计清楚
- 功能演示顺畅
- 文档与视频可交付
